# Requirements Document

## Introduction

This spec implements **Phase 3 — Validation, dedup, review queue** from `docs/exercise-generation-plan.md`. Phase 2 shipped the generator core (`packages/ai/src/generate.ts`) and the CLI driver (`packages/db/scripts/generate-exercises.ts`); every produced draft is currently inserted as `review_status = 'auto-approved'` and the only dedup safeguard is the `recentStems` list grown inside a single batch (`packages/ai/src/generation-prompts.ts:36`).

Phase 3 closes three gaps the plan calls out:

1. **Quality gate.** A second Claude call per draft scores ambiguity, level match, grammar-point match, and cultural issues. Each draft routes to `auto-approved` / `flagged` / `rejected` based on the validator output (plan §3.1). Anything `flagged` is invisible to learners until a human approves it; anything `rejected` is dropped before any DB insert. The application read paths already filter on `APPROVED_STATUSES = ['auto-approved', 'manual-approved']` (`infra/lambda/src/lib/exercise-filters.ts:21`, consumed by `infra/lambda/src/routes/exercises.ts` and `sessions.ts`), and the partial index `exercises_pool_lookup_idx` (`packages/db/src/schema/exercises.ts:27-30`) accelerates those filtered queries — the gate itself is enforced by what the writer chooses to insert and what the route handlers choose to query.
2. **Across-batch dedup.** A canonical surface key per draft + a `UNIQUE` partial index in `exercises` makes duplicate inserts a no-op, and the writer retries up to 3× per slot when a duplicate is detected (plan §3.2). Within-batch dedup already exists; this widens it to "no two approved exercises in the same `(language, level, type, grammar point)` cell ever share the same canonical surface."
3. **Review queue CLI.** A small interactive tool (`pnpm review:flagged --lang es --level B1`) walks each `flagged` exercise, prints it, and accepts `[a]pprove / [r]eject / [s]kip` — flipping `review_status` to `manual-approved` / `rejected` (plan §3.3). Until an admin UI exists this is the only surface for reviewing flagged drafts; expected volume is <5% of generated drafts at the validator thresholds in §3.1.

After this phase, the CLI from Phase 2 still runs end-to-end with the same arguments — the validator, dedup retry, and `quality_score` writes are wired in transparently. The pool fills with calibrated, deduped exercises; flagged drafts queue for human review; nothing reaches learners without passing the gate.

Per resolved decision #1 in the plan, the validator uses the **same** `claude-sonnet-4-5` model the generator and evaluator use, so calibration stays consistent across all three Claude paths. Per resolved decision #6, the validator's Claude calls share the single Anthropic API key with the live evaluator; the generator's existing `--concurrency` cap of 3 from Phase 2 already leaves headroom, and the validator runs serially per draft within a cell (no new concurrency knob).

### Out of scope (this phase)

The following are explicitly deferred to later phases of the exercise-generation plan and SHALL NOT be implemented here:

- **Lambda + SQS + EventBridge** — Phase 4. The Phase 2 CLI remains the only trigger; the validator and dedup retry land inside the same CLI.
- **Anthropic Messages Batches API** — Phase 4.2. Validator calls go through the standard `client.messages.create` endpoint exclusively, mirroring the generator and evaluator.
- **Pool-depth API and admin dashboard** — Phase 5. Operators read `generation_jobs` directly until the dashboard ships.
- **Web admin review UI** — `review:flagged` is the only review surface in this phase; a Clerk-gated web view is Phase 5.3.
- **Embedding-based / semantic dedup** — Phase 3 ships *surface* dedup only. The plan §3.2 explicitly defers semantic similarity until repetition is observed in practice.
- **Validator for new exercise types** — only `cloze`, `translation`, `vocab_recall` are validated, matching the generator's supported types from Phase 2. Phase 6 adds new types and their validators together.
- **Adaptive validator thresholds** — the routing rule from plan §3.1 is hard-coded as constants in this phase. Per-language or per-level thresholds wait until enough validator output exists to justify them.
- **Backfill of existing seed exercises through the validator** — the 36 hand-authored seeds are already `manual` (`generation_source = 'manual'`) and stay `auto-approved`; the validator runs only on freshly generated drafts.
- **L2→EN translation validation** — only EN→target translations are validated, mirroring Phase 2's generator scope (resolved decision #2).

## Alignment with Product Vision

The product is positioned (`product.md` §2) as **"what you do between italki sessions"** — a practice app for intermediate learners stuck at the plateau. That positioning depends on a *trustworthy* pool: every exercise the learner sees must target the right grammar point at the right level, with an unambiguous correct answer. Phase 2 made the pool *fillable*; Phase 3 is what makes it *trustworthy*.

This phase delivers on three load-bearing pieces of the product vision and tech strategy:

- **Pedagogical calibration** (`docs/progress-tracking.md` §"Layer 3", `docs/exercise-strategy.md` §"Pre-generated pool"). Mastery updates from `EvaluationResult` flow into a Bayesian model that assumes the exercise is calibrated to its tagged grammar point and CEFR level. A draft that targets the wrong grammar point or sits a level above its tag silently corrupts the model — the validator's `levelMatch` and `grammarPointMatch` checks block exactly that class of error.
- **Cost-controlled pre-generation** (`tech.md` §7, plan §1.5). The validator adds one Claude call per generated draft; per the plan's cost model (plan §5), validator cost is ~$60 against the round-1 generation budget of ~$110, both bounded by the same `--max-cost-usd` cap that Phase 2 introduced. Cost arithmetic continues to live in `packages/ai/src/cost-model.ts` — the validator's calls fold into the existing `tokenUsage` totals.
- **Honest skill-based progress** (`product.md` §2.2). The dedup index prevents the same canonical sentence stem (or vocab target word) from appearing twice in a learner's session, which would otherwise inflate apparent mastery from a single piece of memorized surface.

## Requirements

### Requirement 1 — Validator core API in `packages/ai`

**User Story:** As the CLI (this phase) and the future Lambda (Phase 4), I want a single typed function `validateDraft(client, draft, spec)` that returns a structured quality verdict, so that triggers don't need to know about prompt construction, tool schemas, or routing rules.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/ai/src/validate.ts` SHALL export a function with the signature `validateDraft(client: Anthropic, draft: ExerciseDraft, spec: GenerationSpec): Promise<ValidateDraftResult>` and SHALL re-export it through `packages/ai/src/index.ts` alongside the generator exports.
2. WHEN the codebase is built THEN `packages/ai/src/validate.ts` SHALL export a `ValidationResult` type with the exact fields named in plan §3.1: `qualityScore: number` (0..1), `ambiguous: boolean`, `levelMatch: boolean`, `grammarPointMatch: boolean`, `culturalIssues: string[]`, `flaggedReasons: string[]`. The function's return type `ValidateDraftResult` SHALL be `{ result: ValidationResult; tokenUsage: ClaudeUsageBreakdown }` so callers can fold validator usage into the cell's running totals.
3. WHEN `validateDraft` is called THEN it SHALL invoke `client.messages.create` exactly once with `model = VALIDATION_MODEL` (a constant equal to `GENERATION_MODEL` per resolved decision #1), `temperature = 0.0` (vs. generation's 0.7 — strict review needs determinism), `max_tokens = VALIDATION_MAX_TOKENS = 1024`, a single cached `system` block (`cache_control: { type: 'ephemeral' }`), one user message describing the draft, exactly one `tools` entry (`VALIDATION_TOOL`), and `tool_choice: { type: 'tool', name: VALIDATION_TOOL_NAME }`.
4. WHEN Claude returns a tool-use block THEN the function SHALL parse it via `parseValidationResult(input)` (exported), which validates every field type, every numeric range (`qualityScore` in `[0,1]`), and that `culturalIssues` and `flaggedReasons` are arrays of strings; on any shape error it SHALL throw an Error whose message identifies the offending field. Mirrors `parseEvaluationResult` (`packages/ai/src/evaluate.ts:128-200`).
5. WHEN Claude returns no tool-use block, returns the wrong tool name, or returns malformed input THEN `validateDraft` SHALL throw — partial results SHALL NOT be returned silently. The error message SHALL include `stop_reason` (mirror of the no-tool-use throw at `packages/ai/src/evaluate.ts:257-263`) and, for the wrong-tool-name path, the unexpected tool name (mirror of `evaluate.ts:265-269`).
6. WHEN `validateDraft` returns THEN `tokenUsage` SHALL equal the sum reported in `response.usage` across the four tiers (`input_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`, `output_tokens`), each defaulting to `0` when the SDK omits the field. The breakdown SHALL be addable to a generator-cell breakdown via the existing `addUsage(a, b)` from `packages/ai/src/cost-model.ts`.
7. WHEN the validator's tool schema is built THEN `VALIDATION_TOOL` SHALL define `input_schema` as a `type: 'object'` with `properties` for every field on `ValidationResult`, mark `qualityScore`, `ambiguous`, `levelMatch`, `grammarPointMatch`, `culturalIssues`, `flaggedReasons` as `required`, and document each property in plain English so Claude can produce calibrated answers. The tool name SHALL be `submit_validation_result`.
8. WHEN `validateDraft` is called with `draft.contentJson.type` outside `cloze | translation | vocab_recall` THEN it SHALL throw before any Claude request is sent. The guard SHALL key off `draft.contentJson.type` (not `spec.exerciseType`) so a caller that hands the validator a draft whose content type doesn't match the spec is caught by this check independently of spec validation. (This is the seam Phase 6 widens.)
9. WHEN `validateDraft` resolves successfully THEN it SHALL NOT mutate `draft` or `spec` — both inputs are treated as immutable, matching the existing pure-function discipline in `packages/ai`.

### Requirement 2 — Validation prompts and per-type user prompts

**User Story:** As the validator, I want one shared system prompt that loads the rubric and the curriculum context, plus one user-prompt builder per supported exercise type, so that adding a new type in Phase 6 is a localized change.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/ai/src/validation-prompts.ts` SHALL export:
   - `VALIDATION_SYSTEM_PROMPT_TEMPLATE: string` — the cached system prompt template.
   - `buildValidationSystemPrompt(spec: GenerationSpec): string` — interpolates `spec.language`, `spec.cefrLevel`, `spec.grammarPoint.name`, `spec.grammarPoint.description`, `spec.grammarPoint.examplesPositive`, `spec.grammarPoint.commonErrors`, and the full `CEFR_LEVEL_DESCRIPTORS` block (imported from `./prompts.js`). Two calls with the same `spec` SHALL produce byte-identical strings (pure / cache-friendly).
   - `buildValidationUserPrompt(draft: ExerciseDraft): string` — branches on `draft.contentJson.type` and renders the draft's content for review.
2. WHEN the system prompt is built THEN it SHALL contain explicit instructions framing the model as a **strict reviewer**: "Reject anything ambiguous", "treat the draft as suspicious until proven calibrated", "score conservatively". This is the temperature-paired counterpart to the generator's "produce diverse, creative drafts" framing.
3. WHEN the system prompt is built THEN it SHALL list every dimension the validator must score, mapped one-to-one to a `ValidationResult` field, and SHALL describe the routing implication in plain English ("a `qualityScore` below 0.5 will reject the draft; flagging at 0.5–0.7 still routes it to manual review"). This is what gives Claude the context to assign self-consistent scores rather than uniformly high ones.
4. WHEN the system prompt is built THEN it SHALL include the same `CEFR_LEVEL_DESCRIPTORS` block consumed by the generator and evaluator (`packages/ai/src/prompts.ts:23-30`) — there is one source of truth across all three Claude paths. No new constant is introduced for descriptors in this file.
5. WHEN the user prompt is built for a `cloze` draft THEN it SHALL render `instructions`, `sentence`, `correctAnswer`, `options` (when present), and `context` (when present) — every field the evaluator's user prompt renders (`prompts.ts:93-112`). The validator's prompt MAY add a "spec" preamble naming the target language, level, and grammar point so the validator can independently judge `levelMatch` and `grammarPointMatch`.
6. WHEN the user prompt is built for a `translation` draft THEN it SHALL render `instructions`, `sourceText`, `sourceLanguage`, `targetLanguage`, and `referenceTranslation`. WHEN the user prompt is built for a `vocab_recall` draft THEN it SHALL render `instructions`, `prompt`, `expectedWord`, `hints`, and `exampleSentence`.
7. WHEN the codebase is built THEN `validate.ts` SHALL pass the system prompt to `client.messages.create` as a single `system` block with `cache_control: { type: 'ephemeral' }` — matching the cache pattern in the evaluator and generator. The system prompt is identical for every draft in a cell (it interpolates only `spec`-derived fields, not draft-level fields), so the first call in a cell is the cache-write and subsequent calls are cache reads.
8. WHEN `tailRecentStems` and `canonicalSurface` already exist in `generation-prompts.ts` THEN this phase SHALL NOT duplicate them — `canonicalSurface` is reused unchanged from Phase 2 to derive the dedup key (Requirement 4).

### Requirement 3 — Routing rule and review-status assignment

**User Story:** As the CLI's writer, I want the validator's verdict mapped to a `review_status` value through a single named function, so that the threshold constants live in one place and tests can pin them.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/db/scripts/generate-exercises-validate.ts` SHALL export:
   - `VALIDATION_THRESHOLDS = Object.freeze({ approveQualityFloor: 0.7, flagQualityFloor: 0.5 })` — the two threshold constants from plan §3.1.
   - `routeValidationResult(result: ValidationResult): { reviewStatus: ReviewStatus; flaggedReasons: string[] }` — pure function implementing the table below.
2. WHEN `routeValidationResult` is called with `result.qualityScore < VALIDATION_THRESHOLDS.flagQualityFloor` OR `result.culturalIssues.length > 0` THEN it SHALL return `reviewStatus = 'rejected'` and `flaggedReasons` containing every entry of `result.culturalIssues` plus, when applicable, a `'low quality score (<0.5)'` string. **Intentional from plan §3.1**: a high-quality draft (e.g. `qualityScore = 0.9`) with even a single non-empty `culturalIssues` entry SHALL still route to `'rejected'`, not to `'flagged'` — cultural issues are treated as a hard veto rather than a reviewable concern, since the human reviewer in Requirement 6 sees only flagged items and would otherwise never have the chance to consider rejecting culturally problematic drafts.
3. WHEN `routeValidationResult` is called with `result.qualityScore >= VALIDATION_THRESHOLDS.approveQualityFloor` AND `!result.ambiguous` AND `result.levelMatch` AND `result.grammarPointMatch` AND `result.culturalIssues.length === 0` THEN it SHALL return `reviewStatus = 'auto-approved'` and `flaggedReasons = []`.
4. WHEN `routeValidationResult` is called and none of the rules in 3.2 / 3.3 fire THEN it SHALL return `reviewStatus = 'flagged'` and `flaggedReasons` listing **every** failed condition: `'ambiguous'` when `result.ambiguous`, `'level mismatch'` when `!result.levelMatch`, `'grammar point mismatch'` when `!result.grammarPointMatch`, `'low quality score (<0.7)'` when `0.5 <= qualityScore < 0.7`, plus every entry of `result.flaggedReasons`. Order SHALL be deterministic so test assertions are stable.
5. WHEN the codebase is built THEN `ReviewStatus` SHALL be a TypeScript union type `'auto-approved' | 'flagged' | 'rejected' | 'manual-approved'` exported from `generate-exercises-validate.ts`. (`'manual-approved'` is the sole possible source of a non-auto approval; it is set by the review CLI in Requirement 6, not by `routeValidationResult`.)
6. WHEN a draft routes to `'rejected'` THEN it SHALL NOT be inserted into `exercises` at all. WHEN a draft routes to `'flagged'` THEN it SHALL be inserted with `review_status = 'flagged'` and `flagged_reasons` set; the existing application read-path filter (`APPROVED_STATUSES` in `infra/lambda/src/lib/exercise-filters.ts:21`, applied by the Lambda routes that hydrate exercises for learners) keeps it invisible until Requirement 6's CLI flips it to `'manual-approved'`. The schema does not enforce this on its own — the partial index in `packages/db/src/schema/exercises.ts:26-30` only accelerates the queries that include the predicate.
7. WHEN a draft routes to any non-rejected status THEN the inserted row SHALL set `quality_score = result.qualityScore` (the column from Phase 1's migration). The Phase 2 writer leaves this column NULL for now; Phase 3 starts populating it.

### Requirement 4 — Across-batch surface dedup

**User Story:** As an operator filling a cell over many runs (e.g. bumping `--batch-seed` weekly), I want duplicate canonical surfaces silently dropped instead of clogging the pool, so that depth metrics are honest and learners never see the same sentence stem twice.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/db/src/schema/exercises.ts` SHALL declare a unique partial index `exercises_dedup_idx` over `(language, type, difficulty, grammar_point_key, (content_json->>'_dedupKey'))` filtered by `WHERE review_status IN ('auto-approved', 'manual-approved', 'flagged')`. (The `flagged` row is included so that the human reviewer cannot manually approve a flagged duplicate of an already-approved draft — the constraint blocks the approval path before it can corrupt the pool.)
2. WHEN `pnpm db:migrate` is run against a Neon dev branch THEN a new migration `packages/db/migrations/0006_*.sql` SHALL apply the new index without touching any other column or table. The migration SHALL be additive only (forward-only, per `tech.md` §5); it SHALL NOT drop or rename existing indexes.
3. WHEN the writer prepares a draft for insert THEN it SHALL compute `dedupKey = canonicalSurface(draft.contentJson)` (reusing the Phase 2 helper from `packages/ai/src/generation-prompts.ts`) and SHALL set `contentJson._dedupKey = dedupKey` before the INSERT. The discriminator-union `ExerciseContent` from `@language-drill/shared` is **not** modified — the underscore-prefixed field is treated as writer metadata that the partial index reads without the application's type guards caring.
4. WHEN the writer issues `INSERT … ON CONFLICT DO NOTHING` and the partial index rejects a duplicate THEN the writer SHALL detect the silent drop (the `RETURNING { id }` row count being smaller than the input batch) and SHALL retry that ordinal up to **3 times** with a regenerated draft. Each retry SHALL be issued via a fresh `generateBatch(client, retrySpec)` call where `retrySpec` is a clone of the original `GenerationSpec` with `count: 1` and `batchSeed: \`${spec.batchSeed}::retry-N\`` (N = 1, 2, 3). The bumped seed feeds `exerciseDraftId(retrySpec, 0)` (`packages/ai/src/generate.ts:476-487`) so each retry yields a fresh deterministic UUID; the `count: 1` ensures the retry produces exactly one draft to check.
5. WHEN all 3 retries for an ordinal collide THEN the writer SHALL stop retrying that ordinal, count it in `CellResult.dedupGivenUpCount`, and continue with the remaining ordinals — the cell is not failed.
6. WHEN a retry is dispatched THEN the second-pass validator SHALL run on the regenerated draft as well — the retry path goes through the same gate as the original.
7. WHEN a retry's regenerated draft is validated and routes to `'rejected'` THEN the writer SHALL count it as a rejection (Requirement 5) and move to the next retry attempt. Up to 3 *insert attempts* per ordinal — Claude calls inside `generateBatch` are not capped beyond this.
8. WHEN `canonicalSurface(draft.contentJson)` is called for the same `draft` twice THEN it SHALL return the same string (pure). The Phase 2 `normaliseSurface` (`packages/ai/src/generation-prompts.ts:144-149`) currently lowercases + NFKD + diacritic-strips; this phase SHALL extend it to also collapse multiple whitespace runs to a single space and trim leading/trailing whitespace — otherwise trivial whitespace differences would defeat the dedup index. The change SHALL be added to the existing `normaliseSurface` helper so every existing caller (`canonicalSurface` in `generation-prompts.ts:151-166`, the `recentStems`-population loop in `packages/ai/src/generate.ts:593-596`, the `inBatchDuplicate` marker logic) inherits the new behavior consistently. Phase 2's existing test file `packages/ai/src/generation-prompts.test.ts` SHALL be extended with whitespace-folding cases as required by Requirement 7.8.

### Requirement 5 — `runOneCell` integration: validator + retry loop + audit-row updates

**User Story:** As the CLI operator, I want the existing `runOneCell` workflow to transparently absorb validation, dedup-retry, and the per-cell audit-row writes, so that `pnpm generate:exercises …` runs the full pipeline without new flags.

#### Acceptance Criteria

1. WHEN `runOneCell` finishes a `generateBatch` call THEN it SHALL iterate the resulting drafts serially, calling `validateDraft(client, draft, spec)` once per draft, folding the validator's `tokenUsage` into the cell's running `tokenUsage`, and recording the routed `reviewStatus` per draft.
2. WHEN any draft routes to `'rejected'` THEN the writer SHALL skip the INSERT for that draft and increment a `rejectedCount` accumulator. (The `generation_jobs.rejected_count` column already exists from Phase 1; it now starts being populated.)
3. WHEN any draft routes to `'flagged'` THEN the writer SHALL INSERT the row with `review_status = 'flagged'`, `flagged_reasons = <jsonb array>`, `quality_score`, `model_id = GENERATION_MODEL`, and the rest of the columns identical to the `'auto-approved'` path. WHEN any draft routes to `'auto-approved'` THEN it SHALL INSERT with `review_status = 'auto-approved'` and `flagged_reasons = NULL`.
4. WHEN a duplicate-dedup INSERT silently drops a draft (Requirement 4.4) THEN `runOneCell` SHALL invoke the retry path: regenerate the single draft via `generateBatch(client, retrySpec)` with `retrySpec.count = 1` and `retrySpec.batchSeed = \`${spec.batchSeed}::retry-N\``, validate it, and re-attempt INSERT. Up to 3 retry attempts per ordinal (Requirement 4.5).
5. WHEN `runOneCell` finishes processing all drafts and retries THEN it SHALL update the `generation_jobs` row's count columns: `produced_count` ← every draft Claude generated, including retries that were rejected or duplicate-collided; `approved_count` ← every `'auto-approved'` row inserted; `flagged_count` ← every `'flagged'` row inserted; `rejected_count` ← every routed-rejected + every retry-given-up.
6. WHEN `runOneCell` finishes processing all drafts and retries THEN it SHALL update `input_tokens_used` and `output_tokens_used` to the sum across **all** Claude calls in the cell — generator + validator + each retry's generator and validator calls — using `addUsage` from `packages/ai/src/cost-model.ts`. The breakdown's three input tiers (non-cached, cache-write, cache-read) are aggregated into the single `input_tokens_used` column per Phase 1's schema, matching the Phase 2 writer's pattern at `packages/db/scripts/generate-exercises.ts:266-269`.
7. WHEN `runOneCell` finishes processing all drafts and retries THEN it SHALL recompute `cost_usd_estimate = estimateCostUsd(combinedUsage)` against the combined generator + validator + retry breakdown, formatted to 4 decimal places via `.toFixed(4)` (matches Phase 2's writer).
8. WHEN `runOneCell` returns its `CellResult` THEN the type SHALL extend Phase 2's `CellResult` with: `validatedCount: number` (every draft that was sent to the validator, including retries), `flaggedCount: number`, `rejectedCount: number`, `dedupGivenUpCount: number` (count of ordinals where all 3 retries collided). The existing `insertedCount`, `skippedCount`, `inBatchDuplicateCount` fields stay; `insertedCount` now counts only the rows that survived dedup AND validation.
9. WHEN any single Claude call (generator, validator, or retry) throws THEN the cell-level isolation from Phase 2 (`packages/db/scripts/generate-exercises.ts:223-234`) continues to protect the rest of the run — the `generation_jobs` row is updated to `'failed'` with `error_message`, no partial drafts land, and the next cell is dispatched.
10. WHEN the SIGINT handler from Phase 2 (`packages/db/scripts/generate-exercises.ts:90-91`, installed at `:466-468`) sets `aborted = true` mid-validation THEN no further validator calls SHALL be made for that cell and the cell-level `failClosed` path SHALL run — same semantics as Phase 2's "SIGINT during a Claude call".
11. WHEN the validator detects that a draft has `metadata.inBatchDuplicate === true` (Phase 2 marker) THEN the writer SHALL still send it to the validator and route normally — in-batch duplicates that *also* fail dedup will be caught by Requirement 4.4's index path and trigger a retry. The Phase 2 marker remains a reporting field, not a routing decision.

### Requirement 6 — Review-queue CLI

**User Story:** As the human reviewer, I want a small interactive command that walks `flagged` exercises in a chosen slice and lets me approve or reject each one, so that triaging the queue is bounded by my reading speed, not by tool friction.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/db/scripts/review-flagged.ts` SHALL exist as an executable script wired through `pnpm review:flagged` (added to `packages/db/package.json` under `scripts` and root `package.json` under `scripts` with `dotenv -e .env -- pnpm --filter @language-drill/db review:flagged`). Mirrors the Phase 2 wrapping pattern.
2. WHEN the script runs THEN it SHALL accept these flags via the same `parseArgs`-style helper used for `generate-exercises.ts`: `--lang <ES|DE|TR>` (required), `--level <A1|A2|B1|B2|C1|C2>` (optional — when omitted, walks every level in the language), `--type <cloze|translation|vocab_recall>` (optional — when omitted, walks every type), `--grammar-point <key>` (optional — when omitted, walks every grammar point in the slice), `--limit <N>` (optional, default 20 — caps how many flagged rows are pulled per invocation so reviewers aren't trapped in a marathon).
3. WHEN the script runs THEN it SHALL `SELECT id, language, difficulty, type, grammar_point_key, content_json, quality_score, flagged_reasons, generated_at FROM exercises WHERE review_status = 'flagged' AND <slice filters>` ordered by `generated_at ASC` (oldest first) limited to `--limit`. No write SHALL happen before the user types a key.
4. WHEN a flagged exercise is rendered THEN the script SHALL print: a header line with id + cell key + quality score, the JSON-pretty-printed `content_json` (excluding `_dedupKey`), the `flagged_reasons` array as bullets, and the prompt `[a]pprove / [r]eject / [s]kip / [q]uit > `. Input SHALL be read from stdin one keystroke at a time (Node's `readline` in raw mode) — typing a single letter advances; arrow keys / other input prints "use a/r/s/q" and re-prompts.
5. WHEN the user types `a` THEN the script SHALL `UPDATE exercises SET review_status = 'manual-approved', flagged_reasons = NULL WHERE id = <current> AND review_status = 'flagged'` and print "Approved." The trailing `AND review_status = 'flagged'` SHALL guard against a concurrent state change. WHEN the user types `r` THEN the script SHALL `UPDATE exercises SET review_status = 'rejected' WHERE id = <current> AND review_status = 'flagged'` and print "Rejected." `flagged_reasons` is preserved on rejection so a later audit can see why. WHEN the user types `s` THEN the script SHALL leave the row untouched and advance to the next. WHEN the user types `q` THEN the script SHALL exit cleanly with a summary (counts of approved / rejected / skipped this session).
6. WHEN every queried exercise has been processed (or the user typed `q`) THEN the script SHALL print a final summary: `Reviewed N exercise(s): A approved, R rejected, S skipped`, plus `(M flagged remain in this slice — re-run to continue)` when `M > 0`. The "remaining" count is computed from a fresh `SELECT count(*) WHERE review_status = 'flagged' AND <slice filters>`.
7. WHEN the script is invoked with no arguments OR with an invalid `--lang` value THEN it SHALL print a usage message naming the flags from 6.2 and exit non-zero, mirroring `generate-exercises.ts`'s `parseGenerateArgs` error handling.
8. WHEN the test environment runs the script's `main(argv, stdinFixture)` (test entrypoint) THEN the script SHALL accept an injected stdin source, so integration tests can drive the prompt loop without a TTY. Production callers continue to use real stdin.
9. WHEN the script is invoked with `NODE_ENV=production` and without an explicit `--allow-prod` flag THEN it SHALL refuse to run, mirroring the Phase 2 production guard at `generate-exercises.ts:472-477`. Phase 4's Lambda is the production review path; the CLI is dev/admin only.
10. WHEN the user approves a row whose canonical surface duplicates an already-`auto-approved` (or already-`manual-approved`) row in the same cell THEN the partial UNIQUE index from Requirement 4.1 SHALL reject the UPDATE because two rows would carry the same `(language, type, difficulty, grammar_point_key, _dedupKey)` tuple while both being inside the index's `WHERE` clause. (The `flagged` row is already in the index per Requirement 4.1, so flipping it to `manual-approved` doesn't change set membership — the conflict is with the *other* approved row already present in the cell.) The script SHALL catch the unique-violation Postgres error (SQLSTATE `23505`), print "Cannot approve — duplicate of an existing approved exercise in this cell. Marking rejected instead.", and `UPDATE … SET review_status = 'rejected' WHERE id = <current> AND review_status = 'flagged'`. The reviewer is informed; the pool stays consistent.

## Non-Functional Requirements

### Performance

- **Validator added latency budget.** One additional Claude call per draft. With prompt caching on the system block, the per-draft median latency in a hot cell SHALL stay under **3 seconds** for the validator call (matching the evaluator's empirical envelope at temperature 0). The CLI's per-cell wall time approximately doubles; this is acceptable for a dev-time tool.
- **Validator token budget.** Per plan §5: ~1K input + 200 output per validation, no cache hit on the user message but full cache hit on the system block. Approval-rate-adjusted total for a 50-draft cell ≈ 50 × $0.0012 ≈ $0.06 — folded into the existing `--max-cost-usd` cap.
- **Dedup-retry overhead.** The expected duplicate rate for the dev pool is <2% (per the plan's surface-dedup intuition). The 3-retry cap bounds worst-case overhead at 3× per ordinal × <2% of ordinals ≈ <6% extra wall time and tokens per cell. Acceptable.
- **Index write cost.** The new partial UNIQUE index adds one B-tree write per insert that targets the `WHERE` clause. With ~50 inserts per cell and partial-index filtering excluding the seed rows initially, write amplification is negligible.

### Security

- **Validator API key reuse.** The validator uses the same `ANTHROPIC_API_KEY` as the generator and evaluator (`requireEnv('ANTHROPIC_API_KEY')`). No new secret SHALL be introduced. The Phase 2 secret-handling pattern is unchanged.
- **Review CLI database writes.** The `review-flagged.ts` UPDATE statements SHALL be parameterized via Drizzle (no string interpolation into SQL). The `WHERE id = ? AND review_status = 'flagged'` guard is the sole mechanism preventing cross-state corruption of a row whose status changed concurrently.
- **Production guard.** The review CLI SHALL refuse to run in `NODE_ENV=production` without an explicit `--allow-prod` flag (Requirement 6.9) — same pattern as Phase 2's generator CLI. The intent is that production review goes through Phase 5's web admin surface, not the CLI.

### Reliability

- **Idempotency on retry.** Re-running the same `pnpm generate:exercises …` invocation SHALL be a no-op for rows already inserted in a prior run, AND SHALL re-run validation for any row that was previously inserted as `'auto-approved'` only when explicitly requested via a (future) `--re-validate` flag — Phase 3 does not introduce that flag yet; today's behavior is "re-running the CLI never re-validates an already-inserted row." The deterministic ID + `ON CONFLICT DO NOTHING` pattern from Phase 2 is what makes this true.
- **Partial-failure isolation.** A validator timeout, a malformed Claude response, or a unique-violation error from the dedup index SHALL never leak partial state to the database. Either the `INSERT` succeeds atomically (all columns populated correctly) or the row is not written.
- **Audit-row truthfulness.** Every cell SHALL leave one `generation_jobs` row whose counts (`produced_count` / `approved_count` / `flagged_count` / `rejected_count`) sum to a value greater than or equal to `requested_count` (greater because retries increase `produced_count`). When `status = 'failed'`, the counts SHALL still reflect whatever happened before the failure — partial accounting is not an error, silent loss is.

### Usability

- **One-keystroke review.** The review CLI SHALL accept single-letter input (`a`, `r`, `s`, `q`) without requiring Enter — the prompt pace is bounded by the reviewer's reading speed, not by typing overhead.
- **Failure messages name the offending field.** Validator parse errors and dedup retry-give-up messages SHALL identify the draft's `id` (and the cell key) in the error message, so an operator inspecting `generation_jobs.error_message` can find the offending row immediately.
- **Dry-run extension is out of scope.** Phase 2's `--dry-run` printed estimated generation cost; Phase 3 does NOT extend it to also estimate validator cost, because the validator's per-draft token usage is empirical (it depends on the draft's content) and the dry-run pre-empts content. The `--max-cost-usd` cap is the operator's runtime safety net for combined generator + validator cost.
