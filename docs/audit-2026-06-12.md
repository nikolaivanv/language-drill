# Architecture, Code Quality & Security Audit — 2026-06-12

Scope: full monorepo — Lambda API (`infra/lambda/src`), database layer (`packages/db`),
AI integration (`packages/ai`), web frontend (`apps/web` + `packages/api-client`),
CDK stack and CI/CD (`infra/lib`, `.github/workflows`).

Method: five parallel read-only audits (one per area), with the highest-severity
claims re-verified by hand against the source before inclusion. Findings the
verification pass could not confirm were downgraded or dropped.

**Overall assessment:** the codebase is in good shape for its stage. Authorization is
consistently enforced per-user across all routes, there is no SQL-injection surface
(Drizzle parameterization throughout), LLM output is strictly validated before it
touches the DB, error responses don't leak internals, and the prompt-versioning /
Langfuse observability discipline is unusually mature. The issues below are mostly
scaling cliffs, missing operational guardrails, and a handful of race conditions —
not fundamental design flaws.

## 1. Security

### 1.1 No user-deletion path — High

`infra/lambda/src/routes/webhooks/clerk.ts:47` handles only `user.created`. There is
no `user.deleted` handler, and the initial migration
(`packages/db/migrations/0000_uneven_starjammers.sql:125–161`) sets
`ON DELETE no action` on the user FKs of `playlists`, `spaced_repetition_cards`,
`usage_events`, `user_exercise_history`, and `user_language_profiles`. Even a manual
`DELETE FROM users` would fail on those constraints.

Consequences: deleting a Clerk account leaves all PII-adjacent data (answer history,
language profiles) orphaned forever — a GDPR/right-to-erasure gap, and a problem the
moment anyone other than the author signs up (signup is open).

Note the contrast: the newer read-phase tables (`packages/db/src/schema/read.ts:69,
119, 166, 194`) and `userPreferences` (`schema/users.ts:34`) correctly use
`onDelete: 'cascade'` — the older tables predate that convention.

**Recommendation:** (a) one migration adding `ON DELETE CASCADE` to the five legacy
user FKs; (b) subscribe the Clerk webhook to `user.deleted` and delete the user row
(cascades do the rest). Both are small, independent changes.

> **Resolved (#284):** migration `0021_audit_cascade_and_usage_events_index.sql`
> adds `ON DELETE CASCADE` to all five legacy user FKs;
> `webhooks/clerk.ts` now handles `user.deleted` (deletes the user row, cascades
> sweep dependents). Operational follow-up: subscribe the Clerk dashboard
> webhook to `user.deleted` (noted in CLAUDE.md).

### 1.2 Streaming Lambda Function URL exposure — Medium-High

`infra/lib/constructs/annotate-stream-lambda.ts:126,137` — `authType:
FunctionUrlAuthType.NONE` with `allowedOrigins: ["*"]`. This is a known design
constraint (RESPONSE_STREAM can't sit behind the API Gateway JWT authorizer; the
handler verifies the Clerk JWT itself, and the in-code comment correctly calls CORS
a politeness filter). The JWT check inside the handler is sound.

The residual risk is **unauthenticated invocation cost**: anyone with the URL can
invoke the Lambda; requests are rejected before reaching Claude, but each rejection
is still a billed invocation, and unlike the generation Lambdas
(`generation-lambda.ts:81` caps reserved concurrency at 3) this function has **no
reserved-concurrency cap and no throttle**. A flood can both run up Lambda cost and
exhaust the account's unreserved concurrency pool, starving the main API.

**Recommendation:** set `reservedConcurrentExecutions` on the streaming Lambda (it
also bounds blast radius), add a CloudWatch alarm on its invocation count, and
record the threat model + risk acceptance in `docs/tech.md`. Longer term, CloudFront
in front of the Function URL (with OAC) adds WAF/throttling capability.

> **Resolved (#284):** `reservedConcurrentExecutions: 10` + an invocation-flood
> alarm (>1000/hr → SNS email) on the streaming Lambda; threat model + risk
> acceptance recorded in `docs/tech.md` (§12). CloudFront/WAF left as the
> documented longer-term hardening.

### 1.3 Usage metering: check-then-insert race — Medium

`infra/lambda/src/routes/exercises.ts:197–214` (same pattern in `/read` and the
annotate-stream handler): `SELECT count(*)` of today's events, compare to the limit,
then proceed to the Claude call and `INSERT` the usage event. Concurrent requests at
the boundary both read count = 49 and both pass. Burst-parallel requests can overshoot
the daily cap by the concurrency factor.

**Recommendation:** for a hard guarantee, insert the usage event *first* inside a
transaction with a `SELECT … FOR UPDATE`-style guard, or use an atomic Upstash Redis
`INCR` (already a dependency, and what it's built for). If approximate enforcement is
acceptable at current scale, say so in a comment — the same way the global-capacity
60s cache (`usage/global-capacity.ts`) already documents its drift window.

> **Resolved (#284):** accepted at current scale and documented — a comment at
> all three metering sites (exercises submit, read generate, annotate-stream)
> records the bounded boundary-overshoot, mirroring the global-capacity drift
> note. Revisit with an atomic Redis `INCR` if multi-user load makes the
> overshoot material.

### 1.4 Invite redemption: non-atomic claim — Low-Medium

`infra/lambda/src/routes/invites.ts:72–84` — read invite, check `usedBy`, then two
sequential `UPDATE`s with no transaction and no guard on the write. The in-code
comment explicitly accepts this at current scale, which is fair — but the fix is
nearly free: add `AND used_by IS NULL` to the invite `UPDATE`'s `WHERE` and check
the affected-row count. That closes the two-users-one-code race without a
transaction; keeping both writes in one `db.transaction` also fixes the
"invite consumed but plan not boosted" partial-failure path.

> **Resolved (#284):** redemption now runs both writes in a `db.transaction`;
> the claim is a conditional `UPDATE … WHERE used_by IS NULL` whose affected
> rows are read via `.returning()` — zero rows ⇒ `409` (lost the race), so the
> user is never boosted on a code we didn't consume.

### 1.5 Prompt injection — Low (adequately mitigated)

User answers are interpolated raw into the evaluation user prompt
(`packages/ai/src/prompts.ts`, `**User's Answer:** ${userAnswer}`), but the call
forces structured tool output (`tool_choice: {type: "tool"}` in `evaluate.ts`) and
every field of the result is range/enum-validated in `parseEvaluationResult` before
persistence. A malicious answer can at most flatter its own score — an accepted and
bounded risk for a self-assessment product. **If a free-prose surface is ever added,
wrap user text in explicit delimiters first.**

### 1.6 Frontend — no significant findings

No `dangerouslySetInnerHTML` anywhere; LLM feedback renders through JSX text nodes
(auto-escaped); Zod validates every API response in `packages/api-client`; Clerk
middleware covers all non-public routes; admin pages are gated server-side in
`app/(dashboard)/admin/layout.tsx` *and* the underlying admin API is independently
gated by `ADMIN_USER_IDS` — the client-side page is not a security boundary and
doesn't need to be. Two small hardening items:

- **Raw `error.message` rendered to users** in several places
  (`app/(dashboard)/layout.tsx:45`, `drill/page.tsx:299`,
  `components/invite/redeem-code-box.tsx:55`). Backend messages are currently
  opaque, so this is latent — but mapping known error `code`s to friendly copy
  (with a generic default) removes the dependency on the backend staying terse.
- **`flaggedMap` keyed by LLM-streamed strings**
  (`apps/web/.../read-page-reducer.ts:102`): a `__proto__` key from the stream
  would prototype-pollute the plain object. Use a `Map` or filter the key. Low
  likelihood, one-line fix.

### 1.7 Things verified as done well

- Every route group mounts `authMiddleware`; every query is scoped `eq(table.userId, userId)`; ownership checks return 403/404 consistently (spot-checked sessions, read, review, progress, vocabulary).
- Clerk webhook verifies the svix signature before processing; the route is intentionally authorizer-free (correct — webhooks can't carry user JWTs).
- All raw SQL (`sessions/today`, debrief, progress radar/heatmap) uses Drizzle's `sql` template parameterization — no string concatenation found.
- IAM in CDK is tightly scoped: per-secret, per-bucket grants; no wildcard actions.
- The dev auth bypass (`DEV_USER_ID`) is structurally unreachable in deployed Lambdas (only the dev entrypoints set it; the deployed handlers don't).
- API keys are never logged or serialized; secrets come from Secrets Manager, none hardcoded.

## 2. Database & data model

### 2.1 Missing index on `usage_events` — High

The daily-limit query runs on **every** AI submission
(`infra/lambda/src/routes/exercises.ts:198–207`: filter on `user_id`, `event_type`,
`created_at >= now-24h`), but `usage_events` has **no index at all**
(`packages/db/src/schema/access.ts` defines indexes only for `invitations`; verified
across all migrations). Today that's a sequential scan of an append-only table that
grows with every AI call — the cost of the *limit check* grows with total usage and
sits on the hot path in front of every Claude call.

**Recommendation:** `CREATE INDEX ON usage_events (user_id, event_type, created_at)`.
One migration, no code change.

> **Resolved (#284):** composite index
> `usage_events_user_id_event_type_created_at_idx (user_id, event_type, created_at)`
> added in migration `0021` (and `usageEvents` in `schema/access.ts`).

### 2.2 `ORDER BY random()` for pool selection — Medium

`infra/lambda/src/routes/exercises.ts:85` — `ORDER BY random()` forces a full scan +
sort of the matching pool slice on every fetch and defeats the carefully built
partial index (`exercises_pool_lookup_idx`). Fine at hundreds of rows; a cliff at
tens of thousands.

**Recommendation:** no urgency, but when the pool grows: fetch N candidate ids via
the index and sample in application code, or use `TABLESAMPLE`.

> **Resolved (#284):** `GET /exercises` now fetches candidate ids via the
> index, samples one in application code, then fetches that row by primary key
> — no `ORDER BY random()` full sort.

### 2.3 Schema integrity gaps — Medium

- `packages/db/src/schema/exercises.ts:11–13`: `type`, `language`, `difficulty` are
  nullable, but routes project them into API responses without null handling. Add
  `.notNull()` (backfill first).
- `packages/db/src/schema/progress.ts:11–13`: `userExerciseHistory.userId` /
  `exerciseId` are nullable FKs with no delete behavior — decide `notNull` +
  cascade, or `SET NULL`, but the current ambiguity means history rows can silently
  detach.
- `playlist_items` / `exercise_tags` FKs are `no action` — deleting an exercise from
  the pool will fail once it's tagged or playlisted. Define the retention policy now,
  while the tables are small.

### 2.4 Done well

Read-phase schema is the model to copy: cascades on all user FKs, composite
`(userId, language, dueAt)` indexes matching the queue query, CHECK constraints on
theory enums, unique constraint on invite codes, transactional entry+vocab upsert
(`routes/read.ts:175–234`).

## 3. AI integration

### 3.1 Unbounded user-answer length — Medium

`SubmitAnswerSchema` enforces only a *minimum* answer length; there is no `.max()`
(verified: no max-length anywhere in `routes/exercises.ts`). A 100 KB answer is
~25k input tokens per evaluation — at 50 free evaluations/day per user this is a
real cost-amplification lever, and very long prompts also push against the 18 s
evaluation timeout.

**Recommendation:** `answer: z.string().min(…).max(2000)` (or whatever the longest
legitimate exercise answer warrants) on every surface that forwards user text to
Claude (submit, annotate, annotate-span).

> **Resolved (#284):** `SubmitAnswerSchema.answer` now `.max(EXERCISE_ANSWER_MAX_CHARS)`
> (= 2000, new constant in `packages/shared`). The annotate / annotate-span
> surfaces were already bounded by `READ_TEXT_MAX_CHARS`, so only submit needed it.

### 3.2 Langfuse fallback is near-silent — Medium

`packages/ai/src/prompts-registry.ts` (`fetchOrFallback`): any fetch failure —
timeout, SDK error, *or the prompt being deleted/unlabeled in Langfuse* — silently
serves the in-repo fallback with a single deduplicated `console.warn` per cold
start. The trace cohort tag (`fallback:<version>`) doesn't distinguish "keys not
configured" (expected in some envs) from "production prompt vanished" (operator
emergency). Given that Langfuse is the *live* source and the repo constant may be
stale, a deployment mishap could quietly serve old prompt bodies for days.

**Recommendation:** add a `fallbackReason` (`keys_unset` / `timeout` / `fetch_error`)
to the resolved version tag, and alarm (CloudWatch metric filter on the warn line,
or a Langfuse evaluator) on any non-`keys_unset` fallback in production.

> **Resolved (#284):** `promptFallbackReason` (`keys_unset` / `timeout` /
> `fetch_error`) is now set on the trace context and surfaced into trace
> metadata, plus a stable `[prompts-registry] prompt fallback (reason=…)` warn
> line emitted only on non-`keys_unset` fallbacks. A CloudWatch metric filter +
> alarm on that line (→ SNS email) is wired on both the API and annotate-stream
> log groups. **Note:** the reason was put in a dedicated metadata field rather
> than concatenated into the `fallback:<version>` tag, to keep that documented
> cohort tag (and its pinned tests) stable.

### 3.3 Done well

- Every LLM tool output passes a hand-rolled strict parser (type, range, enum
  checks) before use; malformed output throws rather than degrading — fail-closed.
- Retry/timeout discipline is deliberate and correct per surface: 18 s / 1 retry on
  the interactive path, SDK defaults for SQS batch work, no client timeout on
  streaming.
- Prompt caching is correctly structured (static system block with
  `cache_control: ephemeral`, volatile user content outside the cached block).
- Version-constant discipline (`<surface>@YYYY-MM-DD`) is enforced by tests that
  pin the expected values.

### 3.4 Minor

- Duplicated helpers across prompt files (`renderBulletList` exists in both
  `validation-prompts.ts` and `generation-prompts.ts`); Turkish-morphology guidance
  is maintained independently in `prompts.ts` and `annotate.ts` — extract a shared
  `prompt-utils.ts` / language-guidance module so a rule change can't drift.

## 4. Infrastructure & CI/CD

### 4.1 No billing alarm — High

The stack has good *application-level* cost brakes (`AI_KILL_SWITCH`,
`AI_GLOBAL_DAILY_CAP`) but **no AWS-level cost visibility**: no
`AWS/Billing EstimatedCharges` alarm, no anomaly detection. The kill switch can't
stop a runaway Lambda loop, an S3 misconfiguration, or invocation flooding of the
open Function URL (§1.2). For a solo-operated AI product, the billing alarm is the
single highest-leverage operational addition.

**Recommendation:** one `EstimatedCharges` alarm per stack (plus, ideally, AWS Cost
Anomaly Detection — a console-side one-off). ~15 lines of CDK.

> **Resolved (#284):** implemented as an **AWS Budget** (`CfnBudget`) instead of
> an `EstimatedCharges` CloudWatch alarm — that metric is published only in
> us-east-1, but the stacks deploy to eu-central-1; Budgets are region-agnostic.
> Monthly cost budget with email alerts at 80%/100% actual + 100% forecast,
> created on the **prod stack only** (budgets track account-wide spend, so a
> second would double-count). Cost Anomaly Detection remains the suggested
> console-side add-on.

### 4.2 No log retention on the hot Lambdas — Medium

`infra/lib/constructs/lambda.ts` (main API) and `annotate-stream-lambda.ts` set no
`logRetention`; their log groups default to *never expire*. The generation pipeline
already sets `ONE_MONTH` (`generation-lambda.ts:83`) — the two highest-volume
functions are the only ones missing it. Unbounded CloudWatch storage cost; also,
CLAUDE.md designates CloudWatch as the API error inbox, so these are the logs that
matter.

> **Resolved (#284):** both the API (`lambda.ts`) and annotate-stream Lambdas now
> use an explicit `logs.LogGroup` with `ONE_MONTH` retention (the explicit group
> also backs the §3.2 prompt-fallback metric filter).

### 4.3 Legacy `JobsQueue` DLQ has no alarm — Medium

`infra/lib/constructs/queue.ts` creates a DLQ but exports no depth alarm, while
both generation queues do (`generation-queue.ts:47–59`,
`theory-generation-queue.ts:56–72`). Messages dying on the legacy queue are
invisible. Either add the same alarm or, if the queue is vestigial, delete it.

### 4.4 No post-deploy verification — Medium

`deploy.yml`'s production ordering is correct and properly serialized
(migrate → CDK → Vercel; the job that runs in parallel with `vercel-prod` is the
*dev* migrate, which is harmless). What's missing is any smoke check after deploy —
a broken Lambda bundle goes live silently. Add a final job that curls
`https://api.langdrill.app/health` (with retries) and fails the workflow.

Related, docs-level: forward-only migrations mean breaking schema changes need the
standard two-phase pattern (expand → deploy → contract). Worth a short runbook
(`docs/runbooks/`) before it bites.

### 4.5 Secrets at deploy time — informational

All Lambdas receive secrets via `secretValue.unsafeUnwrap()` into env vars at synth
time — standard CDK practice, but it means **rotating a secret requires a redeploy**
to take effect. Document this in the rotation story; no code change needed unless
rotation becomes frequent (then switch to runtime fetch with the Secrets Manager
Lambda extension).

### 4.6 Done well

- Tight IAM scoping throughout; no wildcards found.
- Generation/theory pipelines are the operational gold standard in this repo:
  reserved concurrency, DLQs *with* depth alarms, `reportBatchItemFailures`,
  explicit log retention.
- CI: minimal `GITHUB_TOKEN` permissions, concurrency groups on both workflows,
  no secret echoing, Neon branch-per-PR.
- Prod/dev isolation via two instantiations of one stack class matches the docs.

## 5. Architecture & code quality

### 5.1 Shared-constant drift risk — Medium

The learning-language enum (ES/DE/TR) is independently re-declared in at least four
backend locations (`routes/sessions.ts:33`, `routes/progress.ts:27`,
`routes/read.ts:50`, `annotate-stream/handler.ts:65`) plus the client mirror in
`packages/api-client/src/schemas/preferences.ts`. Comments acknowledge it and tests
catch drift, but `packages/shared` exists precisely for this. Move the language
list (and CEFR levels, bucket names, limits if not already there) into
`packages/shared` — this matters more once the Expo app consumes the same contracts.

### 5.2 Hotspot files — Low-Medium

- `apps/web/app/(dashboard)/read/page.tsx` — ~1,200 lines, 15+ hooks in one client
  component. Extract the stream/cache orchestration into a hook and split the view
  switch into subcomponents before the next feature lands there.
- `infra/lambda/src/routes/review.ts` — ~690 lines of route handlers doing FSRS
  state transformation inline. The helpers are already factored; pull the handler
  bodies into a service module as the next change there forces a touch.

### 5.3 Smaller items

- **TanStack Query keys are magic strings** duplicated between hooks and pages
  (e.g. `['readEntry', id]`) — centralize in a `queryKeys` object in
  `packages/api-client`; a typo currently breaks cache invalidation silently.
- **No query retry config** on mutations/queries in `packages/api-client` — a
  single transient 5xx surfaces straight to the user. Add bounded retry with
  backoff for idempotent GETs (leave mutations non-retried unless idempotent).
- **Error boundaries** exist only around theory sections; add one at the dashboard
  layout level so a render error doesn't blank the whole app.
- **Path-param UUID validation** is done in some handlers (sessions debrief) but
  not most — standardize a tiny `z.string().uuid()` guard; saves a DB round-trip on
  garbage input.
- **Dead code:** `infra/lambda/src/middleware/invite.ts` (already documented as
  dead in CLAUDE.md) — delete it; dead auth middleware is the worst kind of clutter.
- `routes/me.ts:12` mounts auth as `/me` instead of the `/me/*` pattern used
  elsewhere — functionally fine, normalize for consistency.
