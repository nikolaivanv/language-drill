# Requirements Document

## Introduction

This spec implements **Phase 2 — Generator core + CLI** from `docs/theory-generation-plan.md`. Phase 1 shipped the data foundation: `TheoryTopicJson` taxonomy + `parseTheoryTopicJson` in `@language-drill/shared`, the `renderTheoryTopicJson` renderer in `apps/web`, the `theory_topics` + `theory_generation_jobs` tables (migrations 0008 + 0009), and the `assertValidTheoryCellKey` / `buildTheoryCellKey` helpers. Nothing in Phase 1 calls Claude, contacts the network, or inserts a row.

Phase 2 turns that foundation into a working generator. After this phase, a developer can run a single command from their laptop:

```
pnpm generate:theory --lang es --grammar-point es-b1-present-subjunctive
```

…and the `theory_topics` table gains one deterministic, idempotent, Claude-generated theory page tagged to that grammar point. The generator core is shaped so Phase 4 (Lambda + SQS + EventBridge) can wrap it without core changes, and Phase 3 (validator + routing) can drop in between generation and INSERT without touching the CLI surface.

Per resolved decision #1 in the plan, the output is structured JSON (a `TheoryTopicJson` payload via Claude tool use), not TSX. Per resolved decision #2, the generator uses the same `claude-sonnet-4-5` model the exercise generator and evaluator use today — pinned via the shared `GENERATION_MODEL` constant from `packages/ai/src/generate.ts:47` so the two paths cannot drift on model version. Per resolved decision #4, cells are keyed `(language, grammarPointKey)` with no `exerciseType` segment and no per-cell count: exactly one theory page per cell. Per resolved decision #5, EN is rejected at the CLI's argument parser; theory pages are written in English about the target language. Per resolved decision #6, vocab umbrellas (`grammarPoint.kind === 'vocab'`) are skipped — theory pages are grammar-only in round 1. Per resolved decision #10, the topic id slug is `grammarPoint.key.replace(/^[a-z]{2}-/, '')` (strip language prefix only; keep the level prefix so cross-level points stay distinct). Per resolved decision #12, `parseTheoryTopicJson` is the boundary — bad output is rejected at parse time, not at render time.

### Out of scope (this phase)

These are explicitly deferred and SHALL NOT be implemented here:

- **The validator** — `packages/ai/src/theory-validate.ts`, the `THEORY_VALIDATION_TOOL` schema, validator prompt builders, the routing rules that set `review_status` to `'flagged'` or `'rejected'`, and the validator's `quality_score` writes are Phase 3. Phase 2 inserts every successfully parsed draft as `review_status = 'auto-approved'` so reviewers can see end-to-end behavior; Phase 3 tightens this gate (per plan §4.1, "Call validateTheoryDraft (Phase 3 — Phase 2 ships with auto-approved)").
- **Review CLI** — `pnpm review:flagged-theory` and its parse-args helper are Phase 3.
- **Lambda + SQS + EventBridge** — Phase 4. Phase 2's CLI is the only trigger.
- **Anthropic Messages Batches API** — not used in theory generation; the workload is small (240 cells round-1 per plan §5) and the cost is bounded (~$17 round-1) so the standard `client.messages.create` endpoint is sufficient.
- **Pool-depth API and admin coverage tile** — Phase 5.
- **Panel registry fallback** — Phase 5; the Theory Panel still reads only the static TSX registry through this phase.
- **New block variants on `TheoryBlockJson`** — Phase 1 fixed the taxonomy. Phase 2 emits and parses through the existing union (`paragraph | callout | example | list | conjugation-table`); any new variant requires the three-step extension named in NFR Maintainability of Phase 1.
- **Migration of the three hand-authored ES TSX files into the DB** — explicitly NOT done (Phase 1 resolved decision #11). They remain the panel's editorial-override path; the generator produces only NEW rows for cells without TSX equivalents.
- **EN-language theory pages** — EN is rejected at the CLI boundary, matching the exercise CLI's EN rejection (resolved decision #4 of the exercise plan; resolved decision #5 of the theory plan).
- **Generation for `kind: 'vocab'` curriculum entries** — `enumerateTheoryCells` filters to `kind === 'grammar'`. Vocab umbrellas would need a different page template (the plan reserves this for a future phase, out of scope here).
- **In-batch surface dedup / `recentStems` / `MAX_DEDUP_RETRIES`** — there is no "batch" in theory generation: one cell = one page = one Claude call. No retry loop, no recent-stems prompt segment, no per-ordinal accumulator.

## Alignment with Product Vision

The Theory Panel shipped via `.claude/specs/theory-panel/` is the in-context reference learners reach for when a drill stumps them. Today it carries three hand-authored ES topics; DE and TR have zero coverage. The product positions itself as **"what you do between italki sessions"** for intermediate learners stuck at the plateau (`product.md` §2 / `CLAUDE.md` §Positioning) — and the plateau cohort at B1–B2 will demand the panel work across **all** grammar points the curriculum names, not just the three a human author had time to write.

Phase 2 is the load-bearing piece of the **content scale-out** strategy from `docs/theory-generation-plan.md` because it's the first phase that actually inserts rows. Phase 1 pinned the data shape; Phase 2 fills the table. Specifically:

- **Schema-pinned tool use** (plan §3, "Output: structured JSON, not TSX"). Phase 2's generator emits via Claude tool-use with `input_schema` mirroring `TheoryTopicJson`. The parser shipped in Phase 1 (`parseTheoryTopicJson` in `@language-drill/shared`) is the boundary that rejects bad output before any DB write. This is the gating mechanism that made the exercise generator reliable; Phase 2 reuses it.
- **Deterministic ID + ON CONFLICT DO NOTHING** (plan §1, success criterion 6 — "reproducible"). The same `(language, grammarPointKey, batchSeed)` produces the same UUID. Re-running the CLI on a cell that already has an approved row is a silent no-op — the operator's "skip; this is filled" intent is naturally expressed by the partial unique index Phase 1 shipped.
- **Cost-controlled** (`CLAUDE.md` §"Pre-generated content"). Every batch declares its expected token spend up front via `--dry-run`; hard-capped via `--max-cost-usd` (default $1.00, per plan §5). The per-cell audit row in `theory_generation_jobs` (Phase 1) records token and cost numbers so the existing pattern (operator queries audit tables for "$ spent this week") works without new tooling.
- **Reuse of the exercise generator's framework, not its surface** (plan §3, reuse map). Phase 2 imports `GENERATION_MODEL`, `addUsage`, `estimateCostUsd`, `ZERO_USAGE`, `createClaudeClient`, `deterministicUuid` from `@language-drill/ai` / `@language-drill/shared` so the two generators never drift on model version, cost math, or hash function. What Phase 2 does NOT reuse is named in plan §3 ("What does NOT transfer cleanly"): in-batch dedup, per-cell pool depth, the exerciseType segment, the retry loop. Those collapse because cardinality is exactly 1 per cell.

After Phase 2 lands, the operator can fill the entire ES theory catalog (~20 grammar points across B1+B2 per current curriculum) for roughly **$1.50** at list pricing and verify the result by toggling the Phase 5 panel fallthrough on a dev branch (Phase 5 is out of scope but its read path is unblocked by Phase 2's writes).

## Requirements

### Requirement 1 — `generateTheoryTopic` generator core

**User Story:** As Phase 2's CLI and as Phase 4's Lambda handler, I want a single async function `generateTheoryTopic(client, spec)` that performs one Claude call with tool use, parses the result through `parseTheoryTopicJson`, and returns a typed `TheoryDraft` plus a `ClaudeUsageBreakdown` so that every downstream caller (CLI, scheduler-driven Lambda, future test harness) inherits identical semantics, cost accounting, and error-message format.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/ai/src/theory-generate.ts` SHALL export the type `TheoryGenerationSpec` with fields: `language: Exclude<Language, Language.EN>` (typed narrow), `cefrLevel: CurriculumCefrLevel`, `grammarPoint: GrammarPoint` (only entries with `kind === 'grammar'` permitted — enforced by Requirement 1.7), and `batchSeed: string`. No `exerciseType`, no `count`, no `topicDomain` — those columns don't apply to theory cells.
2. WHEN the codebase is built THEN `packages/ai/src/theory-generate.ts` SHALL export the type `TheoryDraft` with fields: `id: string` (deterministic UUID — Requirement 3.1), `topicId: string` (kebab-case panel slug — Requirement 3.2), `contentJson: TheoryTopicJson`, and `metadata` carrying `grammarPointKey: string`, `modelId: string`, `inputTokens: number`, `outputTokens: number`, `cacheCreationInputTokens: number`, `cacheReadInputTokens: number`.
3. WHEN `generateTheoryTopic(client, spec)` is called THEN it SHALL invoke `client.messages.create` exactly once with `model: GENERATION_MODEL` (imported from `@language-drill/ai`), `tool_choice: { type: 'tool', name: THEORY_TOOL_NAME }` where `THEORY_TOOL_NAME` is exported from the same file and equals `'submit_theory_topic'`, `tools: [THEORY_GENERATION_TOOL]`, `temperature: THEORY_GENERATION_TEMPERATURE` (the constant equals `0.4` — lower than exercise generation's `0.7`; theory needs accuracy over surface diversity), `max_tokens: THEORY_GENERATION_MAX_TOKENS` (the constant equals `8192` — theory pages are 5–8× longer than exercises), and `system` carrying a single text block with `cache_control: { type: 'ephemeral' }`. The user message comes from `buildTheoryUserPrompt(spec)` (Requirement 2.4); the system text comes from `buildTheorySystemPrompt(spec)` (Requirement 2.3).
4. WHEN `generateTheoryTopic` receives the response THEN it SHALL find the `tool_use` block whose `name === THEORY_TOOL_NAME`. IF no matching `tool_use` block is found THEN it SHALL throw `Error('Theory draft malformed: no tool_use block returned (stop_reason=<response.stop_reason>)')`. IF the matching block's `name` is wrong THEN it SHALL throw `Error("Theory draft malformed: expected tool '<THEORY_TOOL_NAME>', got '<actual>'")` — same error-message convention as `generateBatch` in `packages/ai/src/generate.ts:577-581`.
5. WHEN `generateTheoryTopic` parses the tool input THEN it SHALL call `parseTheoryTopicJson(toolUseBlock.input)` (imported from `@language-drill/shared`). IF the parser throws THEN `generateTheoryTopic` SHALL re-throw with the message prefixed `Theory draft malformed: <parser message>`. The parser's path-prefixed messages (Phase 1 Req 2.x — `Invalid sections[2].body[1].rows[3]: …`) MUST surface verbatim in the re-thrown error so the audit-table reader can find the offending field.
6. WHEN `generateTheoryTopic` succeeds THEN it SHALL return `{ draft, tokenUsage }` where `draft.id === theoryDraftId(spec)` (Requirement 3.1), `draft.topicId === deriveTheoryTopicId(spec.grammarPoint.key)` (Requirement 3.2), `draft.contentJson` is the parser-validated `TheoryTopicJson`, and `draft.metadata.modelId === GENERATION_MODEL`. The `tokenUsage` field SHALL be derived from `response.usage` with `inputTokens`, `cacheCreationInputTokens`, `cacheReadInputTokens`, `outputTokens` each defaulted to `0` if the response field is undefined (mirror of `readUsage` in `packages/ai/src/generate.ts:495-503`).
7. IF `spec.grammarPoint.kind !== 'grammar'` THEN `generateTheoryTopic` SHALL throw `Error("Theory generator: grammarPoint kind 'vocab' is not supported in round 1 (resolved decision #6); got '<kind>' on '<grammarPoint.key>'")` at the top of the function, before the Claude call. The CLI rejects vocab umbrellas earlier (Requirement 6.4); this guard is defense-in-depth for direct SDK callers.
8. IF `(spec.language as Language) === Language.EN` THEN `generateTheoryTopic` SHALL throw `Error('language EN is not a learning language for theory generation (resolved decision #5)')` — matching the EN-reject pattern from `generateBatch:520-524`. The static type narrows EN out; this is defense-in-depth.
9. WHEN `generateTheoryTopic` is re-exported THEN `packages/ai/src/index.ts` SHALL re-export `generateTheoryTopic`, the type `TheoryGenerationSpec`, the type `TheoryDraft`, the type `TheoryGenerateResult` (the `{ draft, tokenUsage }` return shape), the function `theoryDraftId`, the function `deriveTheoryTopicId`, the constant `THEORY_TOOL_NAME`, the constant `THEORY_GENERATION_MODEL` (an alias of `GENERATION_MODEL` for parity with the exercise generator's named export — pinned via cross-file equality in tests), the constant `THEORY_GENERATION_TEMPERATURE`, the constant `THEORY_GENERATION_MAX_TOKENS`, and the typed tool schema `THEORY_GENERATION_TOOL`.

### Requirement 2 — Tool schema, system prompt, user prompt

**User Story:** As the Claude API, I want a single tool whose `input_schema` mirrors `TheoryTopicJson` field-for-field and a system prompt that names the exact required sections so that the model emits parseable output on the first call with high reliability.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/ai/src/theory-generate.ts` SHALL export `THEORY_GENERATION_TOOL: Anthropic.Tool` whose `name` is the value of `THEORY_TOOL_NAME` (`'submit_theory_topic'`), whose `description` is `'Submit a complete grammar theory topic for the configured grammar point. Use the exact section structure named in the system prompt.'`, and whose `input_schema` is a JSON Schema object that mirrors `TheoryTopicJson`'s shape: `type: 'object'` with required fields `id`, `title`, `subtitle`, `cefr`, `sections`. `sections` is an array (minItems: 1) of objects with required fields `id`, `title`, `body` where `body` is an array (minItems: 1) of `TheoryBlockJson` objects expressed via a `oneOf` discriminated on `kind`. Each block variant's schema mirrors Phase 1's TS declaration: `paragraph { text }`, `callout { variant?, children }`, `example { target, en, note? }`, `list { items }`, `conjugation-table { head, rows }`. Inline content (`TheoryInlineJson`) is expressed via a recursive `oneOf` (using JSON Schema `$ref` against a top-level `$defs` entry to express the recursion) discriminated on `kind` covering `text | strong | em | hilite | mono` — the leaf `text` variant carrying `text: string`, the four wrapper variants carrying `children: TheoryInlineJson[]`.
2. WHEN the JSON Schema is structurally checked THEN every required field on every variant SHALL be declared `required` in the matching JSON Schema object — verified by a unit test that loads Phase 1's `subjunctive.json` fixture, serializes it, and asserts that the schema's `required` arrays plus block-variant `oneOf` arms cover every field present in the fixture (Requirement 8.2). The verification does not require runtime AJV — a hand-written walk that asserts presence is sufficient and avoids adding a new package dependency.
3. WHEN the codebase is built THEN `packages/ai/src/theory-prompts.ts` SHALL export `buildTheorySystemPrompt(inputs: TheoryPromptInputs): string` where `TheoryPromptInputs` is `{ language: Exclude<Language, Language.EN>; cefrLevel: CurriculumCefrLevel; grammarPoint: GrammarPoint }`. The function SHALL produce a deterministic string (same inputs → byte-identical output — required for Anthropic prompt caching to hit on subsequent calls within the same cell session, though Phase 2's single-call-per-cell shape means cache hits will materialize across cells in a multi-cell run). The prompt body SHALL include:
   - The role declaration naming the target language (`LANGUAGE_NAMES[inputs.language]`) and CEFR level
   - A `## Grammar point context` block carrying `inputs.grammarPoint.description` verbatim
   - A `## Positive examples` bulleted list from `inputs.grammarPoint.examplesPositive` — the prompt instructs the model to reuse them in the `examples` section
   - A `## Common learner errors` bulleted list from `inputs.grammarPoint.commonErrors` — the prompt instructs the model to address each in the `pitfalls` section
   - A `## Required sections` numbered list naming `what is it?`, `when to use it`, `formation`, `examples in context`, `common pitfalls` — exactly five sections in this order
   - A `## Voice` block: "Editorial. Concise. Lowercase headings. Treat the reader as an adult. No padding, no encouragement, no emojis."
   - A `## Output format` block instructing the model to call the `submit_theory_topic` tool exactly once with the structured topic; inline emphasis goes through the inline-node union; no raw HTML or markdown.
4. WHEN the codebase is built THEN `packages/ai/src/theory-prompts.ts` SHALL export `buildTheoryUserPrompt(inputs: TheoryPromptInputs): string` returning `"Produce the theory page for ${inputs.grammarPoint.name} (${inputs.grammarPoint.key}) at CEFR ${inputs.cefrLevel}."` — single line, no padding.
5. WHEN unit-tested THEN `buildTheorySystemPrompt` SHALL produce identical output for identical inputs (deterministic property) AND SHALL include each curriculum field (`description`, every `examplesPositive` entry, every `commonErrors` entry) as a substring of the returned text. The unit test SHALL pick one ES grammar point from the live curriculum (e.g. `es-b1-present-subjunctive` if present, or whichever B1 grammar entry exists in the current curriculum module) and assert each field appears.
6. WHEN unit-tested THEN `buildTheorySystemPrompt` SHALL declare the five required section names in the order: `what is it?`, `when to use it`, `formation`, `examples in context`, `common pitfalls`. The prompt SHALL NOT name other sections (the generator must produce exactly these five — Requirement 1's correctness path depends on it).

### Requirement 3 — Deterministic IDs and topic-id derivation

**User Story:** As an operator re-running the CLI on a cell that already has a row, I want the second run to be a silent no-op (no error, no duplicate, no audit-row spam) so that the CLI is safe to invoke from cron-like schedules or to retry on transient failures.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/ai/src/theory-generate.ts` SHALL export `theoryDraftId(spec: TheoryGenerationSpec): string` returning `deterministicUuid([spec.language, spec.grammarPoint.key, spec.batchSeed].join('|'))` (`deterministicUuid` imported from `@language-drill/shared`). The function SHALL be pure: same `spec` → same UUID byte-for-byte across runs and across processes. Notably it MUST NOT include any ordinal — theory is one page per cell, not N drafts.
2. WHEN the codebase is built THEN `packages/ai/src/theory-generate.ts` SHALL export `deriveTheoryTopicId(grammarPointKey: string): string` returning `grammarPointKey.replace(/^[a-z]{2}-/, '')` — strips the leading language prefix only (resolved decision #10). For `'es-b1-present-subjunctive'` it returns `'b1-present-subjunctive'`. For `'de-a2-akkusativ'` it returns `'a2-akkusativ'`.
3. WHEN `deriveTheoryTopicId` is called with an input that doesn't match `/^(es|de|tr)-(a1|a2|b1|b2)-[a-z0-9-]+$/` THEN it SHALL throw `Error('Invalid grammar point key for topic-id derivation: <key>')`. The curriculum invariants (`packages/db/src/curriculum/index.ts:62-65`) already enforce the key format; this guard catches direct SDK misuse.
4. WHEN the CLI inserts the row produced by `generateTheoryTopic` THEN the INSERT SHALL use `INSERT ... ON CONFLICT DO NOTHING` (`.onConflictDoNothing()` in Drizzle) against the `theory_topics_pool_lookup_idx` partial unique index (Phase 1 Req 4.2). The Drizzle `.returning()` clause SHALL be checked for an empty array — if empty, the cell was already filled and the run reports `insertedCount = 0, skippedCount = 1` for that cell; if non-empty, `insertedCount = 1, skippedCount = 0`.
5. WHEN the same `--lang`, `--grammar-point`, and `--batch-seed` are passed on a second CLI invocation THEN the second run SHALL succeed with exit code 0, SHALL print a `[ES <level> <grammar-point>] 0 inserted, 1 skipped` line for the cell, and SHALL still write a `theory_generation_jobs` row recording the run (status `succeeded`, `approved=false`, `error_message='cell already filled (partial index collision)'` for forensic clarity — see Requirement 4.7 for the exact column semantics).
6. WHEN the operator bumps `--batch-seed` (e.g. from the default `'theory-v1'` to `'theory-v2'`) on a cell that already has an approved row THEN the new run SHALL also be a silent skip — the deterministic ID changes (so the `id`-collision doesn't fire) but the `theory_topics_pool_lookup_idx` collision (which keys on `(language, grammar_point_key)` regardless of `id`) still fires and the partial unique index rejects the second approved row. Bumping the batch seed is only useful for re-generating a previously **flagged** cell that has been re-routed to `rejected` (Phase 3), or for cells that don't yet have any approved row.

### Requirement 4 — Per-cell orchestrator `runOneTheoryCell`

**User Story:** As Phase 2's CLI and as Phase 4's Lambda handler, I want a single async function that takes a typed input bag and runs the full per-cell pipeline (open audit row → generate → INSERT → close audit row) with cell-isolated try/catch, so that the CLI's only job is argument parsing, cell resolution, concurrency, and summary printing — the actual generation is testable in isolation through this function.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/db/src/theory-generation/run-one-cell.ts` SHALL export `runOneTheoryCell(input: RunOneTheoryCellInput): Promise<TheoryCellResult>` where `RunOneTheoryCellInput` carries: `db: Db`, `client: Anthropic`, `cell: TheoryCell` (Requirement 5), `args: { batchSeed: string; maxCostUsd: number }`, `jobId: string` (caller-supplied), `trigger: 'cli' | 'scheduled' | 'admin'`, optional `signal?: AbortSignal`. No `count`, no `topicDomain`, no per-type fields — those don't apply to theory.
2. WHEN the codebase is built THEN `packages/db/src/theory-generation/run-one-cell.ts` SHALL export `TheoryCellResult` with fields: `cell: TheoryCell`, `jobId: string`, `status: 'succeeded' | 'failed' | 'skipped-cost-cap'`, `insertedCount: 0 | 1` (a literal-typed flag — theory is one page per cell), `skippedCount: 0 | 1` (1 when the INSERT collided with the partial unique index — i.e., the cell was already filled), `tokenUsage: ClaudeUsageBreakdown`, `costUsd: number`, `durationMs: number`, optional `errorMessage?: string`. NO `validatedCount`, `flaggedCount`, `rejectedCount`, `dedupGivenUpCount` — those collapse because Phase 2 has no validator and no in-batch dedup (Phase 3 reintroduces the validation-result counts).
3. WHEN `runOneTheoryCell` begins THEN it SHALL call `assertValidTheoryCellKey(input.cell.cellKey)` as defense-in-depth (the cell builder in Requirement 5 already enforces it). IF the assert throws THEN the function SHALL return a `failClosed`-style result with `status: 'failed'`, `errorMessage` set to the assert's message, NO audit row written (the precheck happened before the audit-row INSERT).
4. WHEN `runOneTheoryCell` writes the opening audit row THEN it SHALL INSERT into `theory_generation_jobs` with `id: input.jobId`, `cellKey: input.cell.cellKey`, `status: 'running'`, `trigger: input.trigger`. The unique constraint on `id` (PK) is the idempotency guard — a re-fire of the same `jobId` collides and the INSERT rejects, which `runOneTheoryCell` SHALL catch and translate into `status: 'failed', errorMessage: 'Audit row id collision (job already ran)'` returning without making the Claude call. This protects Phase 4's deterministic-`jobId` scheduler from causing duplicate runs.
5. WHEN `runOneTheoryCell` calls `generateTheoryTopic` THEN it SHALL build the `TheoryGenerationSpec` from the cell + args: `language: input.cell.language`, `cefrLevel: input.cell.cefrLevel`, `grammarPoint: input.cell.grammarPoint`, `batchSeed: input.args.batchSeed`. The token usage from the call SHALL be accumulated into the cell-result's `tokenUsage` before any later step (so a failure on INSERT still records the cost paid for the Claude call).
6. WHEN `runOneTheoryCell` inserts the row THEN it SHALL INSERT into `theory_topics` with: `id: draft.id` (from `theoryDraftId`), `language: input.cell.language`, `grammarPointKey: input.cell.grammarPoint.key`, `topicId: draft.topicId`, `cefrLevel: input.cell.cefrLevel`, `contentJson: draft.contentJson`, `generationSource: 'claude-realtime'`, `modelId: GENERATION_MODEL`, `reviewStatus: 'auto-approved'` (Phase 2 has no validator — Phase 3 will route here), `qualityScore: null`, `flaggedReasons: null`, `generatedAt: new Date()` (the cell-run timestamp). The INSERT SHALL use `.onConflictDoNothing()` and the `.returning({ id: theoryTopics.id })` array SHALL be inspected — empty means the partial unique index rejected (cell already filled).
7. WHEN `runOneTheoryCell` closes the audit row THEN it SHALL UPDATE the row with terminal status. On success path: `status: 'succeeded'`, `finishedAt: now`, `inputTokensUsed: usage.inputTokens + usage.cacheCreationInputTokens + usage.cacheReadInputTokens`, `outputTokensUsed: usage.outputTokens`, `costUsdEstimate: cost.toFixed(4)`, `approved: insertedCount === 1`, `flagged: false`, `rejected: false`. On dedup-skip path (INSERT returned empty): same as success path EXCEPT `approved: false` AND `errorMessage: 'cell already filled (partial index collision)'`. On Claude-failure path: `failClosed` updates with `status: 'failed', finishedAt: now, errorMessage` truncated to 1000 chars (matching the `ERROR_MESSAGE_MAX_LENGTH` constant from the exercise orchestrator).
8. WHEN the optional `signal: AbortSignal` is aborted at any point during `runOneTheoryCell` THEN the function SHALL throw `Error('Aborted by user (SIGINT)')` — same message as the exercise generator's SIGINT bridge (`run-one-cell.ts:149`). The CLI's SIGINT handler bridges to an `AbortController`; the Lambda passes `undefined`. The cell-isolated try/catch SHALL convert this throw into the cell's `TheoryCellResult` with `status: 'failed', errorMessage: 'Aborted by user (SIGINT)'`.
9. WHEN `runOneTheoryCell` returns with `status: 'failed'` and an `errorMessage` THEN the audit row's `error_message` column SHALL contain the same message truncated to 1000 chars. The CLI's summary printer SHALL show the first 80 chars on the per-cell line (matching `ERROR_MESSAGE_HEAD_LENGTH` from `generate-exercises.ts:62`).
10. WHEN the codebase is built THEN `packages/db/src/theory-generation/index.ts` SHALL re-export `runOneTheoryCell`, the `RunOneTheoryCellInput` type, the `TheoryCellResult` type, plus the `TheoryCell` type and `enumerateTheoryCells` function from Requirement 5. The `@language-drill/db` package barrel (`packages/db/src/index.ts`) SHALL append `export * from './theory-generation';` so callers can `import { runOneTheoryCell, enumerateTheoryCells, type TheoryCell } from '@language-drill/db'`.

### Requirement 5 — `enumerateTheoryCells` and the `TheoryCell` type

**User Story:** As the CLI's cell resolver and Phase 4's scheduler, I want a single canonical builder that turns the curriculum into the universe of theory cells so that the CLI and scheduler cannot drift on which cells exist as the curriculum evolves.

#### Acceptance Criteria

1. WHEN the codebase is built THEN `packages/db/src/theory-generation/cells.ts` SHALL export the type `TheoryCell` carrying: `language: LearningLanguage`, `cefrLevel: CurriculumCefrLevel`, `grammarPoint: GrammarPoint` (the curriculum entry verbatim — keeps the prompt builders' single source of truth), `cellKey: string` (the canonical `<lang>:<level>:<grammar_point_key>` string from `buildTheoryCellKey`). No `exerciseType` field — theory has no per-type fan-out.
2. WHEN the codebase is built THEN `packages/db/src/theory-generation/cells.ts` SHALL export `enumerateTheoryCells(curriculum: readonly GrammarPoint[]): TheoryCell[]` — a pure function (no DB, no I/O) that returns one cell per `GrammarPoint` whose `kind === 'grammar'`. Vocab umbrellas (`kind === 'vocab'`) are silently filtered out — they don't render as theory pages in round 1 (resolved decision #6). The returned array's order matches the input curriculum's order.
3. WHEN `enumerateTheoryCells` constructs each cell THEN it SHALL call `buildTheoryCellKey({ language: entry.language, cefrLevel: entry.cefrLevel, grammarPointKey: entry.key })` and assign the result to `cellKey`. The `buildTheoryCellKey` helper (Phase 1 Req 7) lowercases inputs before joining AND calls `assertValidTheoryCellKey` on its own output, so a curriculum entry that ever drifted from the key format would surface here at enumeration time rather than at INSERT time.
4. WHEN the codebase is built THEN `packages/db/src/theory-generation/cells.ts` SHALL export `THEORY_ROUND_1_CEFR_LEVELS` as a re-export of `ROUND_1_CEFR_LEVELS` from `packages/db/src/generation/cells.ts` (the exercise generator's existing constant). Phase 2 of theory generation targets the same A1–B2 round-1 scope; sharing the constant pins the two generators' level scope together.
5. WHEN unit-tested THEN `enumerateTheoryCells` SHALL produce exactly N cells for a curriculum with M `kind: 'grammar'` entries and any number of `kind: 'vocab'` entries, where `N === M` (vocab umbrellas are silently dropped). The test SHALL assert this against the live curriculum (`ALL_CURRICULA`) AND against a synthetic 2-grammar + 2-vocab input to prove the filter works regardless of curriculum content.

### Requirement 6 — CLI driver `pnpm generate:theory`

**User Story:** As a developer wanting to fill the theory catalog for a language, I want a single command that resolves cells, dispatches each through `runOneTheoryCell` with bounded concurrency, applies a cost cap, supports a dry-run, and prints a per-cell summary plus totals — so that the local generation workflow mirrors the exercise generator's UX byte-for-byte minus the bits that don't apply.

#### Acceptance Criteria

1. WHEN `packages/db/scripts/generate-theory.ts` exists THEN running `npx tsx packages/db/scripts/generate-theory.ts --help` SHALL print a help block listing every flag, every flag's range/default, the env vars required (`ANTHROPIC_API_KEY`, `DATABASE_URL`; `MOCK_CLAUDE=1` to substitute the fixture mock), and one example invocation. The help text SHALL exit code 0.
2. WHEN `packages/db/scripts/generate-theory-parse-args.ts` exists THEN it SHALL export `parseTheoryGenerateArgs(argv: readonly string[]): ParsedTheoryArgs` where `ParsedTheoryArgs` carries:
   - `lang: LearningLanguage` (required; rejects EN)
   - `level: CurriculumCefrLevel | 'all'` (optional; default `'all'`)
   - `grammarPoint: string | null` (optional)
   - `batchSeed: string` (optional; default `'theory-v1'`)
   - `maxCostUsd: number` (optional; default `1.0` per plan §5; must be > 0)
   - `concurrency: number` (optional; default `1`; range `[1, 5]`)
   - `dryRun: boolean` (optional; default false)
   - `allowProd: boolean` (optional; default false; required when `NODE_ENV=production`)
   NO `--type`, NO `--count`, NO `--queue`, NO `--topic-domain` flags — those don't apply to theory in Phase 2. (The `--queue` flag will ship in Phase 4 alongside the theory generation Lambda; deliberately omitted here.)
3. WHEN the CLI is invoked with `--lang en` THEN `parseTheoryGenerateArgs` SHALL throw with a message naming the rejection (`"--lang en is not a learning language for theory generation (resolved decision #5). Use es | de | tr."`). The check matches the exercise CLI's EN reject (`generate-exercises-parse-args.ts:115-119`).
4. WHEN the CLI is invoked with a `--grammar-point` value that doesn't match any curriculum entry THEN `resolveTheoryCells` SHALL throw `Error("--grammar-point '<key>' not in curriculum")`. WHEN the matching curriculum entry's `language` doesn't equal `--lang` OR its `cefrLevel` doesn't equal `--level` (if `--level` was specified other than `'all'`) THEN it SHALL throw with a message naming the mismatch. WHEN the matching entry has `kind === 'vocab'` THEN it SHALL throw `Error("--grammar-point '<key>' is a vocab umbrella (kind 'vocab'); theory generation supports only grammar points in round 1 (resolved decision #6)")`.
5. WHEN the CLI is invoked without `--grammar-point` but with `--lang` and `--level` THEN `resolveTheoryCells` SHALL filter the `enumerateTheoryCells(ALL_CURRICULA)` output to cells matching both `language` and `cefrLevel`. WHEN `--level` is `'all'` (the default) THEN the filter applies only on language.
6. WHEN `resolveTheoryCells` produces zero cells for the requested filters THEN it SHALL throw `Error("no cells resolved for --lang <X> --level <Y>" + grammar-point if any)` — same shape as `generate-exercises-resolve-cells.ts:104-108`.
7. WHEN the CLI is invoked with `--dry-run` THEN it SHALL print one line per resolved cell with the cell key, an estimated input-token and output-token count, and an estimated cost (using empirical per-cell numbers: `~5000 input tokens, ~3000 output tokens, ~$0.05 per cell` — based on plan §5's $0.05 generation estimate; the validator's $0.02 top-up does not apply to Phase 2). The dry-run SHALL print a total estimated cost and exit code 0 without making any Claude or DB call.
8. WHEN the CLI is invoked without `--dry-run` and not in `MOCK_CLAUDE=1` mode THEN it SHALL require `ANTHROPIC_API_KEY` and `DATABASE_URL` in the environment, instantiate a Claude client via `createClaudeClient`, a DB client via `createDb`, and dispatch each cell to `runOneTheoryCell` through a `pLimit(args.concurrency)` limiter. The `pLimit` implementation SHALL be lifted byte-identically from `generate-exercises.ts:108-135` (the inline implementation already exists and is tested) — copy or refactor as appropriate so the two CLIs share the helper without producing two implementations.
9. WHEN the CLI's running total cost meets or exceeds `args.maxCostUsd` THEN cells dispatched AFTER the cap is crossed SHALL be returned as `status: 'skipped-cost-cap'` without making any Claude or DB call. Cells already in flight SHALL run to completion (their cost is committed). The check happens inside the `runWithCostCap` wrapper, mirroring `generate-exercises.ts:337-372`.
10. WHEN the CLI catches `SIGINT` THEN it SHALL signal the `AbortController` it created at startup; cells in flight see `signal.aborted === true` at the next checkpoint and throw `Error('Aborted by user (SIGINT)')`; cells dispatched after the signal SHALL be returned as `status: 'skipped-cost-cap'` with `errorMessage: 'Aborted by user (SIGINT)'`. The CLI SHALL exit code 1 if `signal.aborted === true` at the end of the run.
11. WHEN the CLI prints its summary THEN it SHALL emit one line per cell with format `[<lang> <level> <grammar-point-key>] <inserted>/<total> inserted (<skipped> skipped) — <input> input (<cached> cached) / <output> output tokens — $<cost> — <duration> — <status>` followed by a totals block: `Cells: N (X succeeded, Y failed, Z skipped)`, `Topics inserted: <count>`, `Total input tokens / Total output tokens / Total cached`, `Estimated cost: $<total>`, `Total runtime: <duration>`. The summary printer SHALL mirror `generate-exercises.ts:151-256` with the validation-breakdown columns dropped (no `approvedCount`/`flaggedCount`/`rejectedCount`/`dedupGivenUpCount` — those collapse).
12. WHEN the CLI exits THEN exit code SHALL be 0 if all cells succeeded; 1 if any cell failed OR if the total cost exceeded `--max-cost-usd` OR if SIGINT was received. Matches the exercise CLI's exit-code contract (`generate-exercises.ts:402-406`).
13. WHEN the CLI is run in `NODE_ENV=production` without `--allow-prod` THEN it SHALL print `Refusing to run in production. Pass --allow-prod or use the Phase 4 Lambda path.` and exit code 1. The check mirrors `generate-exercises.ts:265-270`. (Phase 4 introduces the Lambda path for theory.)
14. WHEN `package.json` is read THEN it SHALL include a top-level `"generate:theory"` script equivalent to `dotenv -e .env -- pnpm --filter @language-drill/db generate:theory` (mirror of the existing `generate:exercises` line). `packages/db/package.json` SHALL include the matching workspace script `"generate:theory": "npx tsx scripts/generate-theory.ts"`.

## Non-Functional Requirements

### Performance

- `generateTheoryTopic` SHALL complete one Claude call within the Anthropic SDK's default timeout (60s). The `theory-generate.ts` module SHALL NOT impose its own timeout; the SDK's default is the upper bound.
- The CLI's `pLimit` concurrency cap (Req 6.2 default 1, max 5) SHALL be the only rate-limiting mechanism in Phase 2 — Phase 4 introduces Lambda reserved concurrency and the rate-limit budget shared with the live evaluator.
- The system prompt SHALL be marked `cache_control: { type: 'ephemeral' }` so multi-cell runs amortize the system-prompt cost across cells (per `tech.md` §"Prompt caching" — ~80% cost reduction on cache hits). Phase 2's single-call-per-cell shape means the first cell pays the cache-write cost; subsequent cells in the same run hit the cache for system-prompt tokens. NOTE: each cell's system prompt differs (the curriculum fields are injected) so cache hits are realistic only when re-running the **same** cell within the 5-minute ephemeral window — the dominant cache-hit case in Phase 4 will be the scheduler's same-cell retries, not the multi-cell single-run path.

### Reliability

- `parseTheoryTopicJson` is the boundary (Phase 1 contract). `generateTheoryTopic` SHALL NOT accept any malformed output by silently downgrading or coercing — the function fails closed with a path-prefixed message Phase 4's Lambda can write to its CloudWatch logs verbatim.
- `runOneTheoryCell` SHALL be cell-isolated. A single bad cell SHALL NOT halt a multi-cell run — the cell-level try/catch converts any error into `status: 'failed'` and the CLI's `Promise.all` proceeds to the next cell.
- The audit row's `id` collision check (Req 4.4) SHALL be the idempotency guarantee. Re-firing the same `jobId` SHALL never produce a second Claude call or a second row anywhere — this protects Phase 4's deterministic-`jobId` scheduler from causing duplicate spend.
- The partial unique index on `theory_topics_pool_lookup_idx` (Phase 1 Req 4.2) SHALL be the dedup guarantee at the data layer. The generator does not maintain its own dedup state — Phase 1's schema is the source of truth.

### Security

- `ANTHROPIC_API_KEY` and `DATABASE_URL` SHALL be read only from process environment (never from CLI flags, never from a config file). The CLI SHALL exit 1 with a clear error if either is missing on a non-`--dry-run`, non-`MOCK_CLAUDE` run.
- The CLI SHALL refuse to run with `NODE_ENV=production` without `--allow-prod` (Req 6.13) — the supported prod path is Phase 4's Lambda.
- The generator SHALL NOT log the raw Claude response or the `contentJson` to stdout/stderr in normal operation. The summary printer reports cell-level outcomes only. Failure paths SHALL truncate `errorMessage` to 1000 chars before writing to the audit table.
- `MOCK_CLAUDE=1` SHALL be honored only outside production. The Lambda's deployment environment (Phase 4) MUST NOT carry this env var; the CLI's prod guard already blocks the mock path through `--allow-prod` not being set.

### Maintainability

- Phase 2 reuses `GENERATION_MODEL`, `addUsage`, `estimateCostUsd`, `ZERO_USAGE`, `createClaudeClient`, `deterministicUuid` from upstream packages (`@language-drill/ai`, `@language-drill/shared`) — pinned via cross-file equality tests so the theory and exercise generators cannot drift on these (Req 8.3 for model; the cost-model imports are wholesale, no per-side duplication).
- The directory structure mirrors exercise generation: `packages/ai/src/theory-*.ts` for the generator + prompts; `packages/db/src/theory-generation/` for the orchestrator + cells; `packages/db/scripts/generate-theory*.ts` for the CLI. A reader who has navigated the exercise generator SHALL find the theory generator at structurally analogous paths.
- All tests are co-located with the module they test (CLAUDE.md §Testing). No orphan test directories.
- `pnpm lint && pnpm typecheck && pnpm test` from the repo root SHALL be green at PR time (CLAUDE.md §Pre-Push Checks). No exceptions.

### Usability

- The CLI's `--help` text SHALL be self-contained: a developer reading it SHALL be able to invoke the CLI correctly without consulting this spec.
- The summary printer SHALL surface the cost paid for the run as the last line before exit, making the cost-per-run visible without grep'ing the audit table.
- Error messages SHALL include the cell key, the grammar-point key, and the failure reason — so an operator can copy-paste a single line from the summary into a re-run command.
