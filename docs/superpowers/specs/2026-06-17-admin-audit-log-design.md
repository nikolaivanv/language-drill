# Admin Audit Log (Design)

**Status:** approved · **Date:** 2026-06-17 · **Scope:** Tier 2 item #10 (write side only)

Derived from `docs/admin-panel.md` (Tier 2, item 10: "`admin_audit_log` wired into every
mutating action above" + the "Admin audit log" cross-cutting foundation). Adds an append-only
audit table and records every mutating admin action shipped so far. A read-only **viewer** is
explicitly out of scope (small follow-up).

## Goal

Every mutating admin action appends a row to a new `admin_audit_log` table, so there's a
durable trail of who did what to which entity. This is cheap to add now and expensive to
backfill later; it's also the prerequisite for ever sharing the admin panel with a non-author
admin.

## Background (verified against current code)

- **Schema layout** (`packages/db/src/schema/`): one file per domain (`access.ts` holds
  `invitations` + `usageEvents`), re-exported from `schema/index.ts`. New tables follow the
  `access.ts` style (`pgTable`, column helpers, an index map). `drizzle.config.ts` points
  `schema` at `./src/schema/index.ts`, `out` at `./migrations`.
- **Migrations**: generated by `pnpm --filter @language-drill/db db:generate` (`drizzle-kit
  generate`), applied by `pnpm --filter @language-drill/db db:migrate` (`tsx scripts/migrate.ts`).
  Latest is `0025_real_starhawk.sql`; the next is **0026**. The `migrations/meta` journal +
  per-migration snapshot are updated by `db:generate`.
- **DB handle in routes** (`infra/lambda/src/db.ts`): `export const db = createDb(DATABASE_URL)`
  — a Drizzle/Neon-serverless (WebSocket Pool) instance; `.insert(table).values(...)` is the
  insert pattern (see `POST /admin/invites`). `.transaction()` exists but isn't needed here.
- **Mutating routes** (`infra/lambda/src/routes/admin.ts`, all under `authMiddleware +
  adminMiddleware`; acting admin via `c.get('userId')`):
  - Flagged: `POST /admin/flagged/{exercises,theory}/:id/{approve,reject}` — registration loop;
    handler calls `resolveExerciseFlagged`/`resolveTheoryFlagged(id, action)` → returns
    `'approved' | 'rejected' | 'demoted' | 'not_found' | 'already_resolved'`.
  - Content: `POST /admin/content/{exercises,theory}/:id/{demote,reject}` — loop; handler calls
    `transitionContentExercise`/`transitionContentTheory(id, toStatus)` → returns
    `'demoted' | 'rejected' | 'not_found' | 'already_resolved'`.
  - `POST /admin/generate` — validates + resolves the cell, in-flight 409, enqueues an SQS
    `trigger:'admin'` job, returns `{ jobId, status:'queued' }`.
  - `POST /admin/invites` — inserts N invite rows, returns the codes.
  - `POST /admin/invites/:id/revoke` — 404 if missing, 409 if used, no-op 200 if already
    revoked, else updates `revokedAt`.
- **Test mock** (`admin.test.ts`): the `../db` mock captures `.values(rows)` into
  `insertedValuesByTable[tableSentinel]`; tables are mocked in the `@language-drill/db` mock as
  `{ __mock: '<name>', <col>: {...} }` sentinels. A new table needs an `adminAuditLog` sentinel
  added there to be assertable.

## Table — `packages/db/src/schema/audit.ts`

```ts
export const adminAuditLog = pgTable(
  'admin_audit_log',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    adminUserId: text('admin_user_id').notNull(),     // Clerk id of the acting admin
    action: text('action').notNull(),                 // AdminAuditAction string
    targetType: text('target_type').notNull(),         // AdminAuditTargetType string
    targetId: text('target_id'),                       // nullable (invite.create has none)
    metadata: jsonb('metadata'),                       // outcome / count / jobId / reason
    createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  },
  (table) => ({
    createdAtIdx: index('admin_audit_log_created_at_idx').on(table.createdAt),
  }),
);
```
Exported from `schema/index.ts`. Append-only; no FK on `adminUserId` (admins may be env-listed
IDs without a `users` row, and the trail should survive user deletion).

Migration **0026** generated by `db:generate` (commit the `.sql` + `meta` snapshot). **Do NOT
run `db:migrate` locally** — the local `.env` points at the shared Neon `dev` branch, and
applying an unmerged migration there pollutes per-PR CI forks (which fork from `dev`). CI
applies 0026 on the PR's ephemeral branch and on merge to production.

## Helper — `infra/lambda/src/lib/admin-audit.ts`

```ts
export type AdminAuditAction =
  | 'flagged.approve' | 'flagged.reject'
  | 'content.demote'  | 'content.reject'
  | 'generation.trigger'
  | 'invite.create'   | 'invite.revoke';

export type AdminAuditTargetType = 'exercise' | 'theory_topic' | 'cell' | 'invite';

export type AdminAuditEntry = {
  adminUserId: string;
  action: AdminAuditAction;
  targetType: AdminAuditTargetType;
  targetId: string | null;
  metadata?: Record<string, unknown>;
};

// Best-effort: an audit-write failure must never fail an already-succeeded admin action.
export async function recordAdminAction(db: Db, entry: AdminAuditEntry): Promise<void> {
  try {
    await db.insert(adminAuditLog).values({
      adminUserId: entry.adminUserId,
      action: entry.action,
      targetType: entry.targetType,
      targetId: entry.targetId,
      metadata: entry.metadata ?? null,
    });
  } catch (err) {
    console.warn('admin audit log insert failed', { action: entry.action, targetId: entry.targetId, err });
  }
}
```
`Db` is `@language-drill/db`'s exported `Db` type. The helper swallows errors so call sites are
a single `await recordAdminAction(db, {...})` with no per-call try/catch.

## Wiring (`admin.ts`) — record only effective mutations

In each handler, after the action succeeds, call `recordAdminAction`. Record **only when a
real state change happened**; skip `not_found` / `already_resolved` / validation-rejected /
409 paths.

| Handler | Record when | action | targetType | targetId | metadata |
|---|---|---|---|---|---|
| flagged approve | outcome ∈ {approved, rejected, demoted} | `flagged.approve` | `exercise`\|`theory_topic` | id | `{ outcome }` |
| flagged reject | outcome ∈ {approved, rejected, demoted} | `flagged.reject` | `exercise`\|`theory_topic` | id | `{ outcome }` |
| content demote | outcome ∈ {demoted, rejected} | `content.demote` | `exercise`\|`theory_topic` | id | `{ outcome }` |
| content reject | outcome ∈ {demoted, rejected} | `content.reject` | `exercise`\|`theory_topic` | id | `{ outcome }` |
| `/admin/generate` | after SQS enqueue succeeds | `generation.trigger` | `cell` | cellKey | `{ count, jobId }` |
| `/admin/invites` | after insert | `invite.create` | `invite` | `null` | `{ count }` |
| `/admin/invites/:id/revoke` | only on the branch that actually sets `revokedAt` | `invite.revoke` | `invite` | id | `{}` |

`targetType` for flagged/content derives from the loop's `kind` (`'exercises'` →
`'exercise'`, `'theory'` → `'theory_topic'`). The `action` derives from the loop's `action`
var (`approve`/`reject` → `flagged.*`; `demote`/`reject` → `content.*`). Outcome captures
what actually happened (e.g. an `approve` that hits the dedup conflict records
`action:'flagged.approve', metadata:{outcome:'demoted'}`).

## Testing

- **db** (`packages/db/src/schema/audit.test.ts`): assert `adminAuditLog` is exported with the
  expected column set (mirrors the lightweight existing schema tests).
- **lambda** (`admin.test.ts`):
  - Add an `adminAuditLog` sentinel to the `@language-drill/db` mock so inserts are captured in
    `insertedValuesByTable.adminAuditLog`.
  - For representative wired routes, assert on success that an audit row was captured with the
    correct `action` / `targetType` / `targetId` (e.g. flagged approve → exercise; generate →
    cell + cellKey; invite revoke → invite + id).
  - Assert **no** audit row is captured on a non-effective path (e.g. flagged approve returning
    `already_resolved`; generate returning 409; invite revoke 404). Note: staging order — the
    audit insert is the LAST DB op on the success path, so its `.values()` capture reflects the
    action; on non-effective paths the route returns before recording.
  - A unit test for `recordAdminAction` (`infra/lambda/src/lib/admin-audit.test.ts`): inserts on
    the happy path (capture values); swallows + warns when the insert rejects (does not throw).
- Gate: `pnpm lint`, `pnpm typecheck`, `pnpm turbo run test --concurrency=1`.

## Out of scope

- **Audit viewer** — `GET /admin/audit` + a nav page (small follow-up once there's data).
- Retention/pruning (the doc's open decision; defaults to keep-forever — volume is low).
- Future mutating actions (plan management, kill-switch toggle) — they call the same helper
  when built.
- Backfilling actions taken before this ships.

## Risks / notes

- **Migration numbering** — if another migration merges to `main` before this PR, regenerate so
  0026 doesn't collide (take `main`'s `migrations/meta`, `git rm` the stale `.sql`,
  `db:generate`). Low risk for a solo author, but verify before merge.
- **Best-effort means audit gaps are possible** if the insert fails. Acceptable: the action's
  success is what matters to the user; a transactional approach can't cover `/admin/generate`
  (SQS, not a DB write) anyway, so uniform best-effort is the honest model. Failures are logged
  to CloudWatch via `console.warn`.
- **Effective-only recording** means rejected attempts (`not_found`, `already_resolved`, 409)
  are not logged. That's intentional — the log is a record of state changes, not of attempts.
- **No FK on `adminUserId`** — keeps the trail intact across right-to-erasure user deletion and
  for env-listed admins without a `users` row.
